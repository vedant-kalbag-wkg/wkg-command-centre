## Vercel alias / deployment hygiene — ⚠ open

Vercel's auto-promote to the canonical `wkg-command-centre.vercel.app` domain is **not actually firing** on pushes to main, despite the domain being set to "Production" in the project settings dashboard. Three times today after a merge the canonical alias stayed pinned to the previous production deployment; each time it needed a manual `npx vercel alias set <latest> wkg-command-centre.vercel.app` to promote the current main's build.

Cleaned up four stale aliases from the old project name `wkg-kiosk-tool-*`. Preview-branch aliases (`*-git-<branch>-*`) are Vercel-managed and left alone.

**Suggested next step:** add a minimal GitHub Action that runs on push to main, polls `vercel ls` until the latest Production deployment is Ready, then runs `vercel alias set <url> wkg-command-centre.vercel.app`. Requires `VERCEL_TOKEN` repo secret. Rough sketch:

```yaml
name: Promote canonical alias
on: { push: { branches: [main] } }
jobs:
  promote:
    runs-on: ubuntu-latest
    steps:
      - run: |
          for i in {1..30}; do
            DEPLOY=$(npx -y vercel@latest ls wkg-command-centre --token=$TOKEN | grep -E "Ready.*Production" | head -1 | awk '{print $4}')
            [ -n "$DEPLOY" ] && break
            sleep 10
          done
          npx -y vercel@latest alias set $DEPLOY wkg-command-centre.vercel.app --token=$TOKEN
        env:
          TOKEN: ${{ secrets.VERCEL_TOKEN }}
```

This sidesteps the dashboard config entirely.
