# Phase 1 Milestone 4 — CSV Ingestion (Sales)

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task. Also: superpowers:test-driven-development for Tasks 4.2–4.5.

**Goal:** Admin-driven CSV upload pipeline that parses the data-dashboard sales CSV shape, stages every row with per-row validation, commits valid rows atomically into `salesRecords`, and rejects duplicate uploads by source hash.

**Design reference:** `docs/plans/2026-04-16-kiosk-platform-merge-design.md` section 4, plus the M1-shipped tables `salesImports` / `importStagings` / `salesRecords` (src/db/schema.ts:496–562).

**Branch:** `phase-1/m4-csv`

**Tech Stack:** PapaParse 5 (already in deps, plus `@types/papaparse`), server actions, Drizzle transactions, Next.js App Router admin page.

## Scope decisions (confirmed with human 2026-04-16)

- **Sales only.** This milestone ships the sales CSV pipeline. The existing Monday.com importer at `src/app/(app)/settings/data-import/*` is **archived reference code** — kept in the tree as a reference for any analytics imports that later need to pull Monday.com-derived data, but not actively maintained. Removal is a separate future cleanup ticket.
- **Synchronous commit** with Drizzle `db.transaction`. No job queue, no SSE progress stream. The data-dashboard implementation used SSE for UX polish — we defer that.
- **Full-row validation before commit.** If any row is `invalid`, the import commits NONE of the rows. This is the human-confirmed strict behavior. Our M1 schema's staging table is designed for it. Admins fix-and-reupload rather than chase partial upserts.
- **Source-hash dedupe, not row-level upsert.** `sha256` of the CSV bytes. If an import with the same hash already exists (`staging` or `committed`), reject with a clear error. The M1 `salesRecords.(saleRef, transactionDate)` unique constraint enforces row-level uniqueness at the DB; duplicate CSV bytes are caught at the import level.
- **Dimension resolution is fail-fast, not lenient.** Every row must resolve to an existing `locationId`, `productId`, `providerId` via lookup tables. Unknown dimension values → row flagged `invalid`. No auto-create of locations/products in Phase 1.
- **Route location (confirmed):** nested — `src/app/(app)/settings/data-import/sales/` → `/settings/data-import/sales`. The existing `/settings/data-import` page (Monday.com importer) stays live as archived reference code.

### Shapes

```ts
// Canonical shape coming OUT of the parser, per row.
export type ParsedSalesRow = {
  saleRef: string;
  refNo: string | null;
  transactionDate: string;       // ISO YYYY-MM-DD
  transactionTime: string | null; // HH:MM:SS
  outletCode: string;            // to be resolved against locations
  productName: string;           // to be resolved against products
  providerName: string | null;   // to be resolved against providers
  quantity: number;
  grossAmount: string;           // keep as string (decimal); Drizzle numeric is string-typed
  netAmount: string | null;
  discountCode: string | null;
  discountAmount: string | null;
  bookingFee: string | null;
  saleCommission: string | null;
  currency: string;              // defaults to GBP if blank
  customerCode: string | null;
  customerName: string | null;
};

export type RowValidationError = { field: string; message: string };

export type ParseResult = {
  rows: Array<{
    rowNumber: number;
    raw: Record<string, string>;
    parsed: ParsedSalesRow | null;   // null iff errors.length > 0
    errors: RowValidationError[];
  }>;
  dateRangeStart: string | null;     // ISO; across all VALID rows
  dateRangeEnd: string | null;
  totalRows: number;
  validCount: number;
  invalidCount: number;
};

export function parseSalesCsv(text: string): ParseResult;
export function computeSourceHash(bytes: Uint8Array): string; // sha256 hex
```

### Parsing rules

- Use PapaParse with `{ header: true, skipEmptyLines: 'greedy' }`.
- Column mapping is a lookup table (exported for test visibility) mapping data-dashboard CSV headers (e.g. `Saleref`, `Din`, `Time`, `OutletCode`, `ProductName`, `Quantity`, `Gross`, `Net`, `DiscountCode`, `DiscountAmount`, `BookingFee`, `SaleCommission`, `Currency`, `CustomerCode`, `CustomerName`, `ProviderName`, `RefNo`) to our canonical field names.
- Dates: accept `DD-Mon-YY` (data-dashboard convention) and `YYYY-MM-DD`. Output ISO.
- Times: accept `HH:MM` or `HH:MM:SS`. Output `HH:MM:SS`. Empty → null.
- Numerics: trim, strip thousands separators, keep as string (Drizzle `numeric` is string-typed). Empty → null on optional fields; empty on `grossAmount` → error.
- `currency`: uppercase; empty → default `"GBP"`.
- `quantity`: integer; empty or non-positive → error.

### Row-level validation errors (flagged, not thrown)

- Missing `saleRef`.
- Missing or unparseable `transactionDate`.
- Missing or non-positive `quantity`.
- Missing `grossAmount`.
- Unparseable numeric in any numeric field.

### Signature

```ts
export type ResolvedRow = {
  rowNumber: number;
  locationId: string;
  productId: string;
  providerId: string | null;
} | {
  rowNumber: number;
  errors: RowValidationError[];
};

export async function resolveDimensions(
  db: Database,
  rows: ParsedSalesRow[] & { rowNumber: number }[],
): Promise<ResolvedRow[]>;
```

### Rules

- Batch: run THREE queries across all unique `outletCode` / `productName` / `providerName` values found in input. Build in-memory Maps. Per-row lookup is O(1).
- Unknown `outletCode` → row-level error on `outletCode`. (No auto-create.)
- Unknown `productName` → row-level error on `productName`.
- Unknown `providerName` → row-level error on `providerName` (only if non-null).
- Case-insensitive match for `productName` and `providerName`. Exact match for `outletCode`.

### Server action signatures

```ts
// Step 1: upload + stage. Returns an importId and parsed preview summary.
// Throws if an import with the same sourceHash already exists.
export async function stageImport(formData: FormData): Promise<{
  importId: string;
  totalRows: number;
  validCount: number;
  invalidCount: number;
  dateRangeStart: string | null;
  dateRangeEnd: string | null;
  sampleRows: Array<{ rowNumber: number; parsed: ParsedSalesRow | null; errors: RowValidationError[] }>;
}>;

// Step 2: commit staged rows. All-or-nothing.
// Throws if any staged row is still `invalid`.
export async function commitImport(importId: string): Promise<{ committedRows: number }>;

// Step 3: abandon a staged import without committing (deletes staging rows + marks failed).
export async function cancelImport(importId: string): Promise<void>;
```

### Pipeline internals

**`stageImport`:**
1. `requireRole('admin')`.
2. Read uploaded `File` from FormData as text (`await file.text()`).
3. `computeSourceHash(bytes)`. If `salesImports.sourceHash` already exists, throw with a user-friendly error (`Error("This file has already been uploaded — hash already seen on import <id>")`).
4. `parseSalesCsv(text)`. Compute summary.
5. In a transaction:
   - Insert `salesImports` row (`status: 'staging'`, `filename`, `sourceHash`, `uploadedBy`, `rowCount`, `dateRangeStart/End`).
   - Batch-insert `importStagings` rows (1000 per batch). Each row keeps `rawRow`, `parsedRow` (nullable), `status` (`pending` → set to `valid`/`invalid` in next step), `validationErrors`.
   - Run `resolveDimensions` on parsed rows. Merge resolution errors into each staging row's `validationErrors` and mark final `status`. UPDATE the staging rows with the resolution result.
6. Return summary. NO commit to `salesRecords` yet.
7. Audit: write an `auditLogs` entry (action: `sales_import.stage`, entityType: `sales_import`, entityId: importId).

**`commitImport`:**
1. `requireRole('admin')`.
2. Load `salesImports` row. If not `staging`, throw.
3. Count `importStagings` where `status = 'invalid'` for this importId. If > 0, throw `"Cannot commit: N rows have validation errors"`.
4. In a transaction:
   - Load all `parsedRow` jsonb from staging where `status = 'valid'`, re-resolve dimensionIds (kept in `parsedRow` jsonb for traceability — resolve-on-commit keeps us safe if a dimension was deleted after staging).
   - Batch-insert into `salesRecords` (1000 per batch).
   - Update `importStagings.status = 'committed'` for every row.
   - Update `salesImports.status = 'committed'`.
5. Audit: write `auditLogs` entry (action: `sales_import.commit`, diff: `{ committedRows }`).

**Failure modes handled:**
- Unique-constraint collision on `(saleRef, transactionDate)` during commit → whole tx rolls back; import marked `failed`; error surfaced to UI. Add the specific conflicting row(s) to `salesImports.errors` jsonb for post-mortem.
- Hash collision on upload → `stageImport` throws before touching `importStagings`.

### TDD

Integration tests (testcontainers) covering:
- Happy path: 3-row valid CSV → stage returns correct summary → commit writes 3 sales records → staging marked committed → import marked committed.
- Duplicate upload (same hash) → second `stageImport` throws.
- All-or-nothing: 2 valid + 1 invalid row → `commitImport` throws, no rows in `salesRecords`, import still `staging`.
- Unknown outlet → row `invalid`, commit blocked.
- `cancelImport` → staging rows deleted, import marked `failed`, hash remains recorded (so user gets a fresh hash by editing the file, not by retrying same file).
- Non-admin → `requireRole` throws on all three actions.
- Audit log written on stage and commit.
- Row-level unique-constraint violation on commit (two rows with same `saleRef+date` in one upload) → commit transaction rolls back.

### Shape

```ts
export type SalesSourcePullResult = {
  rows: ParsedSalesRow[];
  dateRangeStart: string | null;
  dateRangeEnd: string | null;
  sourceHash: string;
  sourceLabel: string; // "csv:filename.csv" or "supabase:2026-03-01→2026-03-31"
};

export interface SalesDataSource {
  pull(): Promise<SalesSourcePullResult>;
}

export class CsvFileSource implements SalesDataSource {
  constructor(private readonly file: { name: string; text: () => Promise<string>; bytes: () => Promise<Uint8Array> }) {}
  async pull(): Promise<SalesSourcePullResult>;
}
```

### Rationale

This interface is what lets Phase 3 swap `CsvFileSource` for `SupabaseSource` or `PostgresSource` without touching the commit pipeline. `stageImport` is refactored to take a `SalesDataSource` rather than raw FormData — the server action wraps the FormData file in a `CsvFileSource`.
