## Summary

Phase 4.1 is primarily a correctness and extension phase, not a greenfield build. The Monday.com migration infrastructure (field mapper, actions, import UI) was built in Phase 4 and is fully functional â€” but contains five specific bugs where field mappings are wrong or logic is incorrect. Alongside the bug fixes, the phase adds four location schema fields, two new entity pages (Products, Kiosk Config Groups), sidebar navigation entries, revised table column defaults, and a three-step import flow with board ID persistence.

The most complex new work is the location merge logic (D-09/D-10): hotels whose Monday.com item name contains `[outlet_code]` suffixes must be collapsed into a single location record. The current `extractLocationName` function has no such logic. The dry-run preview must surface conflicts between rows that share a normalised hotel name, requiring a new conflict resolution UI step. The Kiosk Config Groups board (1466686598) is entirely unexplored and requires an exploration step in Wave 0 before its field mapping can be written.

**Primary recommendation:** Fix the five field mapper bugs first (they require targeted changes to `KNOWN_FIELD_MAP` and `mapMondayItemToKiosk`), then add the four schema columns, then build the two new entity pages, and handle the import flow extension last. This ordering ensures the schema is stable before new UI is built against it.

## Common Pitfalls

### Pitfall 1: Monday.com group title vs Region column value
**What goes wrong:** `item.group?.title` in `mapMondayItemToKiosk` returns the Monday.com board group (like "London Hotels"), not the "Region" column value (like "London"). These may match but may not â€” they are separate fields.
**Why it happens:** Initial implementation used group title as a fallback when the Region column wasn't the intended source.
**How to avoid:** Always resolve the Region value from the column_values array using the mappings lookup. The Region column is already in `KNOWN_FIELD_MAP` but is used for `regionGroup` field only â€” the kioskId derivation must also use it.
**Warning signs:** KioskIds that look like board group names rather than region prefixes.

### Pitfall 2: Notes pollution from location-mapped columns
**What goes wrong:** The unmapped collector loop at `mapMondayItemToKiosk:198-209` skips columns with `targetTable !== "kiosk"` BUT still runs `getColumnDisplayValue` and adds to `unmappedValues` for any column where the condition `!mapping || mapping.status === "unmapped" || mapping.targetTable !== "kiosk"` is true. This means location-mapped columns (like Hotel Address, Star Rating) get dumped into kiosk notes.
**Why it happens:** The condition was intended to collect only truly unmapped columns but the third clause (`targetTable !== "kiosk"`) incorrectly captures location-targeted columns.
**How to avoid:** Separate the conditions: skip if mapped to location (`targetTable === "location"`), collect into unmapped only if `mapping === undefined || mapping.status === "unmapped"`.
**Warning signs:** Kiosk notes containing hotel address, room count, star rating etc.

### Pitfall 3: Location merge â€” normalisation timing
**What goes wrong:** If name normalisation (`strip [outlet_code]`) is applied after `expandSplitItems`, items already have their outlet code stripped from the name by `expandSplitItems` (line 530: `name: \`${item.name} [${code}]\``). So normalisation must handle this format correctly.
**Why it happens:** `expandSplitItems` appends ` [code]` to the cloned item names. The merge logic must strip this suffix to find the canonical hotel name.
**How to avoid:** `normaliseLocationName` regex `\s*\[[^\]]+\]$` correctly strips this format. Apply it consistently whenever building the deduplication key.
**Warning signs:** Duplicate location records where names differ only by outlet code suffix.

### Pitfall 4: `listLocations` query doesn't return new schema fields
**What goes wrong:** After adding `region`, `locationGroup`, `internalPoc`, `status` to the schema, the `listLocations` query (line 277 in locations/actions.ts) uses a join query that selects specific columns. The new columns won't appear unless the query is updated.
**Why it happens:** Drizzle select queries are explicit â€” adding columns to the table doesn't auto-include them in existing queries.
**How to avoid:** After schema push, update both the Drizzle query (add new columns to `select`) and the `LocationListItem` TypeScript type.
**Warning signs:** TypeScript errors on `locationData.region` etc. after schema push, or missing data in location table rows.

### Pitfall 5: Kiosk Config Groups board exploration must happen before mapping is written
**What goes wrong:** Board `1466686598` has an unknown column structure. Writing a field mapper for it without first calling `exploreBoardColumns` on it will produce incorrect mappings.
**Why it happens:** The CONTEXT.md explicitly flags this: D-05 says the board needs an exploration step.
**How to avoid:** Wave 0 of the plan must include an exploration/inspection step for this board, or at minimum document that the first import run will require the admin to manually review and confirm mappings for the config groups board.
**Warning signs:** Auto-detected mappings for the config groups board returning all "unmapped" columns.

### Pitfall 6: `drizzle-kit push` required (not migrate)
**What goes wrong:** Running `npx drizzle-kit migrate` hangs indefinitely on this project.
**Why it happens:** Supabase session-mode pooler on port 5432 limits max clients. `migrate` opens a session that blocks (see STATE.md: [Plan 02-01]).
**How to avoid:** Always use `npx drizzle-kit push` after schema changes.
**Warning signs:** Terminal hangs with no output after `drizzle-kit migrate`.
