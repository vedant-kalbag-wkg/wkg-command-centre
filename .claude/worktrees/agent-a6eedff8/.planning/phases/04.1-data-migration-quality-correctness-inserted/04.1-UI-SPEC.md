## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding, badge padding |
| sm | 8px | Compact element spacing, button icon gap |
| md | 16px | Default element spacing, card padding, form field gap |
| lg | 24px | Section padding, panel inner padding |
| xl | 32px | Layout gaps, page content gap |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing |

**Touch target exception:** Icon-only controls (table row actions, sidebar toggle) must have a minimum 44px hit area. This is a **component-level hit-area override** applied via padding or a wrapping clickable element — it is NOT a spacing scale token and must not appear in the token set above. Implement as `min-h-[44px] min-w-[44px]` on the button/trigger element.

Source: Established in Phase 1 — 8-point scale. Touch target rule applied consistently across the app.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#FFFFFF` (`--color-wk-white`) | Page backgrounds, card surfaces, popover backgrounds |
| Secondary (30%) | `#121212` (`--color-wk-graphite`) | Sidebar background, data-dense table rows, foreground text |
| Accent (10%) | `#00A6D3` (`--color-wk-azure`) | Reserved for specific elements only (see below) |
| Destructive | `#F41E56` (`--color-wk-destructive`) | Delete actions, destructive confirmation dialogs |

**Accent (`#00A6D3`) is reserved for:**
- Primary CTA buttons (Import Kiosks, Import Hotels, Import Config Groups, Run Import)
- Active sidebar nav item left-border and background tint (`rgba(0, 166, 211, 0.20)`)
- Import progress bar fill
- Merge conflict "selected" radio button state in dry-run preview
- Focus ring on inputs and selects (`--ring: var(--color-wk-azure)`)
- Board ID "Saved" confirmation badge

Accent must NOT be used on secondary buttons, table row backgrounds, badge labels (except where Azure tint is specified), or body text.

Source: `src/app/globals.css` — `:root` CSS variables and sidebar token assignments.

## Component Inventory

New UI surfaces introduced in this phase:

### 1. Products Tab (sidebar nav item, new page)

**Focal point:** The Products table is the primary visual anchor. The "Add Product" CTA button (Azure, top-right) is the secondary anchor. On empty state, the empty state heading "No products yet" anchors the viewport.

- **Nav entry:** Icon `Package` (Lucide), label "Products", href `/products`
- **Page layout:** AppShell with page title "Products", primary CTA "Add Product" (Button, variant primary)
- **Table:** TanStack Table v8 — columns: Product Name, Provider, Hotels Offering, Hotels Unavailable
- **Row click:** Opens product drill-down — filtered view of per-hotel availability (same page, panel or navigation TBD by executor)
- **CRUD:** Add/edit product via Dialog (shadcn Dialog component), delete via destructive Dialog confirmation
- **Icon-only table row actions:** Edit and Delete icons on each row must include `aria-label` (e.g., `aria-label="Edit product"`, `aria-label="Delete product"`) AND a shadcn Tooltip showing the same label text. Both are required — tooltip for sighted pointer users, aria-label for assistive technology.

### 2. Kiosk Config Groups Tab (sidebar nav item, new page)

**Focal point:** The Kiosk Config Groups table is the primary visual anchor. The group name column (leftmost) is the row-level anchor. There is no primary CTA on this page — the import flow (Data Import page) is the entry point for data.

- **Nav entry:** Icon `Layers` (Lucide), label "Kiosk Config Groups", href `/kiosk-config-groups`
- **Page layout:** AppShell with page title "Kiosk Config Groups", no primary CTA (import-only source)
- **Table:** TanStack Table v8 — columns: Group Name, Product Availability (from import), Hotels (clickable count), Kiosks (clickable count)
- **Clickable counts:** Hotels count navigates to `/locations?kioskConfigGroup={groupId}`, Kiosks count navigates to `/kiosks?kioskConfigGroup={groupId}` — use `buttonVariants({ variant: 'link' })` on a Link element

### 3. Data Import Page — Extended Board ID Inputs

**Focal point:** The "Run Import" primary CTA button is the visual anchor of the Data Import page. The three Board ID input fields are the secondary anchor — they gate the import flow and draw attention first on empty/first-run state. The import progress bar (Azure fill) becomes the focal point during an active import run.

- **Board ID fields:** Three labeled text inputs in the existing Settings card
  - "Assets / Kiosks Board ID" — maps to board `1426737864`
  - "Hotels / Locations Board ID(s)" — comma-separated, maps to live/hotel boards
  - "Kiosk Config Groups Board ID" — maps to board `1466686598`
- **Persistence:** Board IDs auto-save to app settings after first entry; pre-populated on return visits with a "Saved" badge in Azure
- **Import sequence indicator:** Three-step progress display (Step 1: Kiosks → Step 2: Hotels → Step 3: Config Groups) using the existing Progress component or a simple step indicator with numbered badges

### 4. Merge Conflict Resolution UI (dry-run step)

- **Trigger:** Dry-run detects hotels with same name but different `[outlet_code]` suffixes (D-09/D-10)
- **Presentation:** Conflict card per conflicting location — shows conflicting field values side-by-side in a two-column layout (Column A: "Row 1 value", Column B: "Row 2 value")
- **Resolution control:** Radio group per conflicting field — user selects the correct value for each field; unresolvable = block import
- **Blocking behaviour:** "Run Import" button is disabled until all conflicts are resolved; disabled state uses `opacity-50 cursor-not-allowed` per existing pattern
- **Conflict card component:** Use shadcn Card with border `--color-wk-azure-40` tint to distinguish from standard preview rows

## Interaction Patterns

### Sequential Import Flow

The three-step import flow extends the existing 6-state data-import UI. States remain: connect → mapping → preview (with optional conflict resolution) → import → complete. The "import" step displays the three sub-steps sequentially with step labels ("Step 1 of 3 — Kiosks", etc.) and a Progress bar advancing per step.

Individual step imports (D-15) appear as secondary buttons alongside the primary "Run Import" CTA.

### Inline Editing — New Location Fields

Region, locationGroup, internalPoc, and status on location detail follow the existing `InlineEditField` pattern (EditableCell for table rows, inline editable text/select for detail page). No new interaction patterns introduced.

### Products Drill-Down

Clicking a product row in the Products table expands or navigates to a per-hotel breakdown. Executor may implement as either: (a) expandable row with accordion, or (b) navigation to `/products/{productId}` detail page. Either approach must be consistent with the existing kiosk/location detail page pattern.

## Accessibility Contract — Icon-Only Actions

All icon-only interactive elements (table row action buttons, icon-only toggles) across all new surfaces in this phase must satisfy both of the following requirements:

1. `aria-label` attribute on the button element with a clear action description (e.g., `aria-label="Edit product"`, `aria-label="Delete config group"`).
2. shadcn `Tooltip` wrapping the button, with `TooltipContent` matching the `aria-label` text exactly.

This applies to all table row actions on the Products table and Kiosk Config Groups table. No icon-only interactive element may ship without both attributes.
