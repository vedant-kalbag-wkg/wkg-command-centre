<decisions>
## Implementation Decisions

### Kiosk Config Groups Entity
- **D-01:** Kiosk Config Groups are content-based groupings — each group defines what content/products are available on the kiosks in that group
- **D-02:** Each kiosk belongs to exactly one group (FK on kiosks table: `kioskConfigGroupId`). Future: multiple/hierarchical groups — design the FK approach to be replaceable but don't build many-to-many now
- **D-03:** Sidebar tab labeled "Kiosk Config Groups" — table view showing: group name (from Monday.com "Item"), product availability columns (from Products tab), clickable hotel count (navigates to Locations page filtered by group), clickable kiosk count (navigates to Kiosks page filtered by group)
- **D-04:** During import, the Kiosk Config Groups board serves as the source of truth for the total product/content offerings
- **D-05:** Board `1466686598` needs an exploration step before mapping — column structure is unknown

### Products Tab
- **D-06:** Dedicated Products tab in sidebar — master catalog view showing: product name, provider, # hotels offering the product, # hotels where unavailable. Click a product row to see per-hotel breakdown
- **D-07:** Per-hotel product configuration (availability, provider, commission tiers) stays on the location detail page — not moved to Products tab
- **D-08:** Products tab is read/manage for the catalog (CRUD on products table); location detail is where per-hotel config lives

### Location Merge Behavior
- **D-09:** Hotels with same name but different `[outlet_code]` suffixes merge into one location record with separate kiosks per outlet code
- **D-10:** When merging, conflicting field values between rows are flagged during dry-run preview — admin must resolve all conflicts before clicking Import. No surprises during actual import
- **D-11:** Clean slate expected — admin clears existing data before re-importing with the merge fix. No need to detect/merge existing duplicate DB records

### Import Flow
- **D-12:** Admin enters board IDs on the data import page: one for assets/kiosks board, one or more for live/hotel boards (comma-separated), one for Kiosk Config Groups board
- **D-13:** Board IDs saved to app settings after first entry — pre-populated on subsequent imports, editable by admin
- **D-14:** Unified import runs in sequence: kiosks → hotels → Kiosk Config Groups (three steps, ordered)
- **D-15:** Individual import option also available — admin can run just kiosks, just hotels, or just groups independently for flexibility
- **D-16:** Kiosk Config Groups import is the third and final step in the sequential flow

### Field Mapping Fixes (from requirements — no gray areas)
- **D-17:** KioskId derived from Region column value + outlet code (e.g., "London-ABC123") — per MIGR-04
- **D-18:** "Assets" column maps to kiosk `hardwareSerialNumber` — per MIGR-05
- **D-19:** CMS Config imported as the actual status label (not raw JSON) — per MIGR-06
- **D-20:** Kiosk notes contain only genuinely relevant notes — unmapped location fields must NOT be dumped into kiosk notes — per MIGR-07
- **D-21:** "Key Contact" column = WeKnow internal POC (`internalPoc` on location), not hotel contact — per MIGR-08
- **D-22:** Location schema extended with `region`, `locationGroup`, `internalPoc`, `status` fields — per MIGR-11

### Table Display Fixes (from requirements — no gray areas)
- **D-23:** Kiosk table default columns: Asset (hardware serial number), outlet code, venue, region, pipeline stage, CMS config status, install date — per MIGR-12
- **D-24:** Location table default columns: name, hotel group, star rating, rooms, # kiosks, address, sourced by, status, maintenance fee, customer code, key contact name, region, location group, internal POC — per MIGR-13

### Claude's Discretion
- Kiosk Config Groups schema design (exact columns beyond name and group-kiosk FK)
- Products tab layout and drill-down UX
- Merge conflict UI presentation in dry-run preview
- Board ID settings storage mechanism (DB table vs config file)
- Import progress UI for the three-step sequential flow
- Field mapper implementation details for the corrected mappings

</decisions>

<specifics>
## Specific Ideas

- Kiosk Config Groups table: clickable hotel count navigates to `/locations?kioskConfigGroup={groupId}`, clickable kiosk count navigates to `/kiosks?kioskConfigGroup={groupId}` — leverages existing table filter infrastructure
- Kiosk Config Groups board is the source of truth for total product offerings during import — seed the products table from this board
- Merge conflict resolution in dry-run: show conflicting fields side-by-side with radio buttons to pick the correct value per field
- Board IDs saved to settings means the import page can pre-populate and just show a "Re-import" button for subsequent runs
- The user calls them "SSM Groups" internally but the UI label should be "Kiosk Config Groups"

</specifics>
